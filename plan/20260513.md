# Demo4 Follow-up Plan - 2026-05-13

## Context

Demo4 validation has inherited the main solver CPU thread scheduling path, and single-case validation now runs with internal OpenMP parallelism. Two residual issues remain before demo4 can be considered practical and easy to interpret.

## 1. Make Evaluation Results Easier to Read

### Problem

The current evaluation metrics are not intuitive enough for users. Columns such as `RMS abs`, `RMS rel`, `max abs`, `max rel`, `AUC abs`, and `AUC rel` are mathematically useful, but the UI does not explain their meaning or rank the result in a direct engineering way.

The plotted curves are also too thick, so overlapping variants visually collapse into what looks like one line. This makes it hard to see whether refinements differ or converge.

### Desired Direction

- Prioritize direct error indicators, especially maximum error and whether it passes the expected tolerance.
- Keep advanced metrics available, but move them behind clearer labels, tooltips, or a secondary details view.
- Reduce chart line width and improve variant styling so multiple curves remain distinguishable.
- Consider showing a compact summary first: pass/fail, worst variant, max absolute error, max relative error, and runtime.
- Replace opaque variant labels such as `ny_refined_1`, `ny_refined_2`, and `ny_refined_3` with direct Ny values such as `ny_50`, `ny_100`, or `ny_150`, so the refinement level is visible without remembering the multiplier rule.
- Add an actual elapsed-time column to the Validation Progress table. The current columns `case`, `variant`, `t`, `endT`, `progress`, and `status` should include `time (s)` so users can see how long each validation row has already taken or took to finish.

### Acceptance Check

A user should be able to open demo4 Evaluation and quickly answer:

- Which variant is worst?
- How large is the largest error?
- Did refinement improve the result?
- Are multiple plotted curves actually different or overlapping?
- What Ny value does each variant actually use?
- How many seconds did each validation row take?

## 2. Optimize Ny-Refined Validation Runtime by Orders of Magnitude

### Problem

The `ny = 50 * 3` validation path is still far too slow:

```text
iterations_per_second = 566.6
```

This is not acceptable for the intended demo4 workflow. It blocks interactive validation and makes refinement studies feel broken even after the CPU thread scheduling fix.

### Current Observation

The baseline validation speed improved after forcing single-case validation to use internal CPU parallelism, but the higher-Ny case remains bottlenecked. This likely requires deeper optimization than thread scheduling alone.

### Desired Direction

- Profile the `ny = 150` validation case separately from baseline.
- Identify whether the bottleneck is solver kernels, output/checkpoint/dense dump work, memory layout, stability/statistics checks, or validation orchestration overhead.
- Compare demo2 and main solver behavior for the same refined Ny case.
- Look for an optimization path that improves runtime by at least one order of magnitude before adding more UI polish.

### Acceptance Check

Demo4 validation should make the `ny = 50 * 3` case fast enough for routine use, with a clear before/after benchmark recorded in the validation output.
