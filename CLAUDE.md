关于 Edit 工具老出现"截断/匹配失败"的问题，最常见有两类原因。下面给你一套实战处理流程，基本能覆盖 90% 的场景。
一、先搞清楚"截断"到底是哪种
通常是这两种之一：

old_string 在文件里找不到 / 不唯一 → 报 "no match" 或 "not unique"
要替换的内容太长，复制过来时被中间省略号截断了 → 你手里的 old_string 其实和文件里的不一致

第 2 种最常见，尤其是从 Read 输出里复制大段代码时。
二、推荐的处理顺序
1. 先用 Read 重新读取目标行附近的内容
不要靠记忆或之前的 Read 结果，要带 offset / limit 精确读取要改的那几行，确保拿到的是当前磁盘上的真实内容（注意去掉 Read 输出里 行号 + Tab 的前缀，只保留真正的代码）。
2. 缩小 old_string 的范围
不要一次替换 50 行。挑一个唯一且短的锚点（5–15 行就够），用它定位。如果上下文不够独特，往前/后再加 1–2 行让它唯一即可。
3. 注意空白字符

Tab vs 空格混用
行尾是 \n 还是 \r\n（Windows 项目里很常见，你这个 ADR 项目就在 Windows 上）
行尾的尾随空格

这些用肉眼看不出来，但 Edit 是按字节精确匹配的。
4. 改不动就分多次小改
把一次大改拆成若干次 5–10 行的小 Edit，每次都先 Read 验证。
5. 重写整段而不是 Edit
如果同一处要改很多地方，干脆 Read 整个文件 → Write 整个文件覆盖。对小文件（<200 行）这反而更快更稳。
6. 用 replace_all
如果是改一个变量名、字符串常量这类全局重命名，直接 replace_all: true，不用纠结上下文。
三、一个常踩的坑
你贴 old_string 时，不要把 Read 输出里的行号前缀（123\t）一起复制进去 —— Edit 会按字面匹配，带了前缀就永远匹配不上。

## Karpathy 风格项目工作准则

本节来自 `andrej-karpathy-skills` 的核心思想，并按本 ADR 项目做了收敛：目标是减少错误假设、过度抽象、无关改动，以及“看起来完成但没有验证”的情况。

### 1. 编码前先澄清

- 涉及物理模型、输入参数含义、输出格式、数值精度或 WebUI 工作流时，不要默默假设。
- 如果一个需求可能指向 C++ 内核、CUDA 后端、MATLAB 绘图脚本或 WebUI，先说明判断依据；不确定时先问。
- 对会改变结果数值、默认参数、checkpoint 行为、文件命名或批量运行范围的改动，要明确说出影响面。

### 2. 简洁优先

- 用当前需求所需的最小改动解决问题，不为了“以后可能会用”提前加抽象层。
- 优先沿用现有结构：`include/` 声明、`src/` 实现、`input/` 案例文件、`tests/` Catch 测试、`webui/` 本地界面。
- 不引入新的框架、配置格式或构建入口，除非需求本身必须这样做。

### 3. 精准修改

- 只改与任务直接相关的文件；不要顺手整理无关代码、注释、格式、备份目录、生成目录或 demo 副本。
- 保持现有风格：C++17、CMake/Makefile shim、UTF-8/LF，`.bat`/`.cmd` 保持 CRLF。
- 如果发现无关的旧问题或死代码，记录或说明即可；除非任务要求，不要删除。
- 如果自己的改动造成未使用变量、include、函数或脚本入口，要一并清掉。

### 4. 目标驱动和可验证闭环

非琐碎任务先把请求转成可验证目标：

```text
1. 明确要改变的行为 -> 验证: 现有/新增测试或最小复现
2. 实现最小改动 -> 验证: 编译、单测、或指定 case 运行
3. 检查副作用 -> 验证: git diff 只包含必要改动
```

常用验证优先级：

- C++ 内核或参数解析：`cmake --build build --target adr_solver_tests`，然后 `ctest --test-dir build --output-on-failure`。
- 输入文件或运行逻辑：用最小 case 验证，不默认跑全量长耗时案例。
- WebUI：启动本地服务并检查关键交互，不只看静态文件。
- 文档或脚本：检查路径、命令和编码，避免写入 GBK/ANSI。
